# 評論審核隊列 — 批量認領設計

**狀態：** 已實施
**作者：** Wilson（與 PM 團隊）
**日期：** 2026-04-13
**範圍：** 4 人審核團隊 · 每日約 1000+ 條評論 · LLM 生成的論壇種草內容

---

## 1. 問題描述

論壇種草平台每日生成約 1000+ 條 LLM 評論待審核。4 人審核團隊並行工作，需要解決兩個問題：

1. **衝突問題** — 兩位審核員操作同一條評論（重複發佈、浪費精力、衝突）
2. **效率問題** — 審核員掃描其他人正在處理的項目，加上過多的逐條點擊

最初提出了兩種方案：

| 方案 | 權衡 |
|---|---|
| **WebSocket 即時鎖定** | 正確但過度工程化；4 個用戶不需要連接生命週期、重連、狀態同步等複雜度 |
| **建立時自動分配** | 僵化 — 請假/工作量不均會導致隊列卡住；無法自我平衡 |

**兩者都不合適。** 本文檔提出第三條路徑：**批量認領 + TTL 租約**（方案 A）。

---

## 2. 選定方案 — "開始審核" 批量拉取

### 2.1 核心思路

移除逐條「認領」按鈕。替換為**一個主要操作**：點擊**「開始審核」** → 後端原子性認領下一批 N 條（預設 10）最舊的未認領評論到該審核員。這 10 條成為審核員的**個人工作台**。通過/拒絕後，工作台自動補充。

### 2.2 為什麼這個方案勝出

- ✅ **零逐條點擊** — 解決了 UX 摩擦
- ✅ **零衝突風險** — Postgres `SELECT ... FOR UPDATE SKIP LOCKED` 保證原子性不重疊批次，即使 4 位審核員同時點擊
- ✅ **不需要 WebSocket，不需要預分配** — 普通 REST + 10 秒輪詢在此規模足夠
- ✅ **自我平衡** — 如果一位審核員請假，工作留在池中；其他人多拉取即可
- ✅ **TTL 自動釋放** — 如果審核員離開，認領過期後項目自動回到池中
- ✅ **鍵盤優先流程** — `J` 通過 / `K` 拒絕 / `S` 跳過，每日 250 條審核無手腕負擔

## 3. 用戶旅程 — 逐步說明

### 3.1 角色

**Emma**，內容審核員，星期一 09:00。她打開後台。三位隊友（Alex、Ben、Carol）也在線。

### 3.2 逐步流程

#### 步驟 1 — Emma 進入隊列頁面

```
┌──────────────────────────────────────────────────┐
│  Feed 隊列                                         │
│  [待審 78]  [已通過]  [已發佈]  [已拒絕]  [失敗]    │
│                                                   │
│           ┌───────────────────────────┐           │
│           │  ▶ 開始審核 (抽取 10 條)   │           │
│           └───────────────────────────┘           │
│                                                   │
│           我的工作台: 0 條                          │
│           全隊待審: 78 條                           │
│                                                   │
└──────────────────────────────────────────────────┘
```

- 沒有列表。沒有卡片。只有一個大按鈕。
- 她看到「全隊 78 條待審」作為上下文。

#### 步驟 2 — Emma 點擊「開始審核」

後端呼叫：

```http
POST /api/v1/review-queue/claim-batch
Body: { "count": 10 }
```

伺服器原子性執行：

```sql
WITH picked AS (
  SELECT id FROM feeds
  WHERE status = 'pending'
    AND (claimed_by IS NULL OR claim_expires_at < NOW())
  ORDER BY created_at ASC
  LIMIT 10
  FOR UPDATE SKIP LOCKED
)
UPDATE feeds
SET claimed_by = :emma_id,
    claimed_at = NOW(),
    claim_expires_at = NOW() + INTERVAL '30 minutes'
WHERE id IN (SELECT id FROM picked)
RETURNING *;
```

Emma 現在擁有評論 #1-10（最舊的優先）。Alex 同時點擊，拿到 #11-20。**不可能重疊** — `SKIP LOCKED` 是關鍵。

#### 步驟 3 — Emma 的工作台視圖

```
┌──────────────────────────────────────────────────┐
│  工作台 (1 / 10)          本批次 28 分鐘後自動釋放   │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━   │
│                                                   │
│  FQ-202604130635-B9E                              │
│  原帖: 個唱慶功宴被集郵 馬國明「老態」畢現            │
│                                                   │
│  生成回覆:                                          │
│  ┌──────────────────────────────────────────┐    │
│  │ 呢個話題好有趣,我都想分享下自己嘅經驗…       │    │
│  └──────────────────────────────────────────┘    │
│                                                   │
│  [✓ 通過 (J)]  [✗ 拒絕 (K)]  [↷ 跳過 (S)]         │
│                                                   │
│  下一條預覽: FQ-202604130635-1ZP 何鴻燊女兒…        │
└──────────────────────────────────────────────────┘
```

- 聚焦視圖 — 一次一條（也可提供收合列表模式）
- 進度：**1 / 10**（遊戲化批次）
- 認領過期時間可見 — Emma 知道離開會發生什麼
- 底部預覽下一條保持節奏

#### 步驟 4 — Emma 審核並操作

她按 **`J`** 通過：

```http
POST /api/v1/review-queue/:id/approve
```

後端驗證 `claimed_by = emma`，標記 `status = 'approved'`，UI 自動推進到第 2 條。**不刷新頁面。不額外點擊。**

她繼續：`J`、`J`、`K`（拒絕一條差的）、`J`、`S`（跳過一條需要上下文的 — 釋放回池）、`J`、`J`、`J`。

#### 步驟 5 — 工作台快用完了

當 Emma 處理到 **7 / 10**（剩 3 條），出現提示：

```
  工作台剩 3 條 · [補充下一批 10 條]  (或自動補充)
```

點擊拉取 10 條。如果她先完成當前 3 條，短暫看到：

```
  🎉 批次完成！處理了 10 條 · 平均 42 秒/條
  [▶ 抽取下一批 10 條]
```

#### 步驟 6 — Emma 午餐離開

她合上筆記本沒完成批次。**30 分鐘後**，認領過期。未完成的項目自動回到池中。Ben 在 13:00 點「開始審核」拿到了其中一些。**不需要人工干預。**

Emma 回來時，工作台顯示：

```
  ⚠ 上次批次已過期並釋放 · [開始新批次]
```

#### 步驟 7 — 下班

Emma 的儀表板：

```
  今日處理:  247 條
  通過:      198    拒絕: 41    跳過: 8
  平均用時:  38 秒/條
```

## 4. API 設計

### 4.1 Schema 變更

```sql
-- Feed 表新增欄位（複用現有 claimed_by/claimed_at，新增 TTL）
ALTER TABLE feeds
  ADD COLUMN claim_expires_at timestamptz NULL;

CREATE INDEX idx_feeds_claim_pool
  ON feeds (status, claim_expires_at, created_at)
  WHERE status = 'pending';
```

### 4.2 端點

| 方法 | 路徑 | 用途 |
|---|---|---|
| `POST` | `/api/v1/review-queue/claim-batch` | 原子性認領下 N 條 → 工作台 |
| `GET`  | `/api/v1/review-queue/my-workbench` | 返回當前用戶的活躍認領 |
| `POST` | `/api/v1/review-queue/:id/approve` | 通過（需 `claimed_by = 我`） |
| `POST` | `/api/v1/review-queue/:id/reject`  | 拒絕（需 `claimed_by = 我`） |
| `POST` | `/api/v1/review-queue/:id/skip`    | 釋放單條回池 |
| `POST` | `/api/v1/review-queue/extend-claims` | 心跳 — 延長活躍用戶的認領 TTL |
| `GET`  | `/api/v1/review-queue/stats` | 團隊級統計（總待審、中位年齡等） |

### 4.3 關鍵端點詳情

**`POST /api/v1/review-queue/claim-batch`**

```json
// 請求
{ "count": 10 }

// 回應
{
  "claimed": [
    { "id": "FQ-202604130635-B9E", "content": "...", "post_title": "...", ... },
    ...
  ],
  "claimExpiresAt": "2026-04-13T09:30:00Z",
  "remainingInPool": 58
}
```

**並發保證：** 使用 `FOR UPDATE SKIP LOCKED` — 如果 Emma 和 Alex 在完全相同的毫秒點擊，Postgres 串行化並給每人一組**不相交**的行。零衝突。

**`POST /api/v1/review-queue/:id/approve`**

```sql
UPDATE feeds
SET status = 'approved', reviewed_by = :me, reviewed_at = NOW()
WHERE id = :id
  AND claimed_by = :me
  AND status = 'pending';
-- 如果 0 行受影響返回 409 Conflict
```

**`POST /api/v1/review-queue/extend-claims`**（心跳）

前端每 5 分鐘在用戶活躍時呼叫。延長所有 `claimed_by = :me` 項目的 `claim_expires_at`。防止工作中的過早過期。

## 5. 前端設計

### 5.1 頁面

1. **隊列首頁** — 單一「開始審核」按鈕 + 團隊統計
2. **工作台視圖** — 一條聚焦視圖，鍵盤驅動
3. **歷史 Tab** — 已通過 / 已發佈 / 已拒絕 / 失敗（唯讀，供審計）

### 5.2 鍵盤快捷鍵

| 按鍵 | 操作 |
|---|---|
| `J` | 通過 + 下一條 |
| `K` | 拒絕 + 下一條 |
| `S` | 跳過（釋放回池）+ 下一條 |
| `E` | 編輯評論內容後再通過 |
| `O` | 在新標籤頁打開原帖 |
| `Esc` | 退出工作台（認領保持到過期） |

### 5.3 輪詢

- 心跳（`extend-claims`）每 5 分鐘（標籤頁聚焦時）
- 團隊統計每 30 秒刷新
- **不需要 WebSocket。**

## 6. 邊界情況處理

| 場景 | 行為 |
|---|---|
| 審核員中途關閉標籤頁 | 認領 TTL 過期（30 分鐘）→ 項目自動回到池 |
| 審核員想找特定評論 | 搜尋欄（FQ-ID 查找）→ 若未認領則自動認領；若被他人認領則唯讀預覽 |
| 4 位審核員同時點「開始審核」 | `SKIP LOCKED` 確保 4 個不相交批次 |
| 池中項目耗盡 | 「開始審核」按鈕禁用，顯示「暫無待審項目」 |
| 審核員反悔之前的決定 | 已通過 Tab 有「撤回」操作（5 分鐘窗口內） |
| 管理員 / QA 抽查 | 管理員角色可唯讀查看任何已認領項目，不影響認領 |

## 9. 被否決的替代方案（留檔）

### 9.1 WebSocket 即時鎖定
- **否決原因：** 連接生命週期、重連邏輯、黏性會話、狀態調和。4 個用戶不存在的問題需要一週以上工作量。

### 9.2 建立時自動分配
- **否決原因：** 僵化。如果審核員請假或速度慢，預分配的項目閒置。無法自我平衡。需要再平衡子系統來修復租約式認領免費提供的功能。

### 9.3 逐條「認領」按鈕（原始設計）
- **否決原因：** UX 摩擦。每人每天 250 條，逐條認領點擊是純成本。也會出現「我點了認領但別人先點了」的 UI 競態。

### 9.4 樂觀並發（無認領，通過時檢查）
- **否決原因：** 解決了正確性但不解決效率 — 審核員仍會閱讀彼此的項目，浪費腦力。

## 10. 團隊討論開放問題

1. **批次大小** — 10 對此複雜度的評論感覺合適。是否應可配置（5/10/20）？
2. **認領 TTL** — 30 分鐘寬裕但安全。審核員速度快的話考慮 20 分鐘。
3. **跳過行為** — 跳過的項目是否標記「此審核員跳過」以防下次拉取到？還是完全重新循環？
4. **SLA 定義** — 評論是否有最大年齡？（例如「生成後 2 小時內必須發佈以保持論壇討論動量」）。需要產品決策。
5. **通過前編輯** — 審核員可以編輯多少 LLM 輸出？允許完全重寫還是僅微調？

## 11. 同事摘要

> 原本考慮 WebSocket 即時鎖定或建立時自動分配，兩者都過度工程。改用 **Batch Claim + TTL Lease** 方案：
>
> - 審核員點 **「開始審核」** → 後端原子性鎖定 10 條到個人工作台
> - `SELECT ... FOR UPDATE SKIP LOCKED` 保證零重複，4 人同時點也 OK
> - 工作台用鍵盤 J/K/S 衝刺處理，0 次逐條點擊
> - 30 分鐘無操作自動釋放回池 → 不會卡住
> - 普通 REST + 每 5 分鐘心跳續期，**不用 WebSocket**
>
> 核心邏輯：claim 是系統概念，使用者只想「開始工作」。把 claim 隱藏成「開始審核」的副作用就對了。
